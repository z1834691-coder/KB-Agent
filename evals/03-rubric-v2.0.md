# KB-Agent 评测标准 v2.0

> 定位：v2.0 是 KB-Agent 的 harness 方向盘。它保留状态评测 / 行为评测 / 版本三元组机制，但把 v0.1 的大维度评分升级为：子指标独立计分、知识类型化保鲜、split 纪律、Judge 校准、成本门槛和标杆 checklist。

## 0. 版本纪律

每轮评测必须记录三元组：

```text
agent_prompt_version × dataset_version/split × rubric_version
```
还有整体分数、failure patterns、每个 case 的前后输出
v2.0 主数据集：`challenge-dataset-v0.4`。被测 agent 只允许看到 actor-pack，Judge 才能看到 gold。任何把 gold 内容带入 actor 上下文的运行判无效。

## 1. 知识类型与保鲜策略

| 类型 | 定义 | 保鲜策略 | 主路径处理 |
|---|---|---|---|
| `foundational_classic` 奠基经典 | 经典论文、概念源头、长期有效方法 | 不按新旧扣分；考核历史定位、开创点、后续演化 | 保留演化链 |
| `framework_methodology` 框架方法论 | 选型框架、评测方法、产品方法论 | 季度/年度复检，标注适用边界 | 保留核心框架 + 变化说明 |
| `product_version_data` 产品版本与数据 | 模型版本、价格、上下文、API、榜单 | 月检或事件触发；accessed_at 必填 | 主路径只放当前值，历史进 changelog |
| `news_dynamic` 新闻动态 | 发布、融资、活动、传闻、短期讨论 | 周检，过期归档；未核实不得进主文 | 只保留对目标有长期价值的摘要 |

时效性不再等于“越新越好”。Judge 必须先判类型，再按该类型策略评分。

## 2. State Eval 权重

### 建设期权重（当前默认）

| 维度           |  权重 | 理由                             |
| ------------ | --: | ------------------------------ |
| D1 信息质量      | 25% | 事实与源纪律是底线，但建设期还要重建流程           |
| D2 内容价值与目标对齐 | 25% | 用户目标是国内头部 AI 公司 AI PM，选材必须服务目标 |
| D3 结构与组织     | 15% | 支撑可检索、可维护、可演化                  |
| D4 呈现与体验     |  5% | 影响学习效率但不是第一瓶颈                  |
| D5 可维护性与流程   | 25% | 当前最大短板，必须优先提升                  |
| D6 标杆实践采纳率   |  5% | 防止口号化对标，作为 checklist 约束        |

### 稳态期权重

切换条件：连续两轮状态分 >= 7.5，D5 >= 8，回归通过率 100%。

| 维度           |  权重 |
| ------------ | --: |
| D1 信息质量      | 30% |
| D2 内容价值与目标对齐 | 25% |
| D3 结构与组织     | 15% |
| D4 呈现与体验     |  5% |
| D5 可维护性与流程   | 20% |
| D6 标杆实践采纳率   |  5% |

## 3. State Eval 子指标

每个子指标独立 1-10 分，必须附证据。维度分 = 子指标均值，除非本节另写权重。

### D1 · 信息质量

| 子指标 | 定义 | 计分规则 | 证据要求 |
|---|---|---|---|
| D1.1 事实准确性 | 关键事实无已知错误；存疑显式标注 | 1-3 多处错误；4-6 抽样有错误或不确定性未标；7-8 关键事实可靠；9-10 随机抽样零错误且有核验记录 | 文件路径/行、源链接、核验记录 |
| D1.2 信源权威与溯源完整 | 一手源优先；二手源有来源链；引用不失真 | 1-3 大量无源；4-6 有源但元数据弱；7-8 核心条目 source metadata 完整；9-10 可追到一手源和 via 链 | source_url、author、accessed_at、via |
| D1.3 类型化保鲜/经典定位 | 按知识类型应用保鲜策略 | 1-3 不分类型；4-6 部分有日期但策略不清；7-8 大多数条目有 type/保鲜；9-10 当前值、历史值和经典定位完全分离 | type、last_reviewed、stale_after 或 historical_position |
| D1.4 交叉验证与冲突处理 | 争议数字/结论有独立源或待核实标记 | 1-3 争议直接入库；4-6 偶有待核实标记；7-8 关键争议有交叉验证；9-10 冲突有专门记录和复检任务 | ≥2 源或 conflict_note |

10 分锚点：随机抽 20 条关键知识，均能判断类型、追到源、看出当前状态或历史定位；无未标注事实错误。

### D2 · 内容价值与目标对齐

Judge 必须引用 `research/golden-kb-study/00-总纲.md` 中的 SL 判据。

| 子指标 | 定义 | 计分规则 | 证据要求 |
|---|---|---|---|
| D2.1 信息密度 | 单位篇幅内事实、洞察、框架含量高 | 1-3 大段低密度搬运；4-6 有信息但冗余多；7-8 摘要高密度；9-10 每节都有事实/洞察/用法 | 摘要和正文对比、冗余段落证据 |
| D2.2 前沿性/经典性匹配 | 对应知识类型的时间价值处理正确 | 1-3 把旧数据当当前或把经典当过时；4-6 部分匹配；7-8 大多数类型处理正确；9-10 各类型策略稳定执行 | 类型 + 保鲜策略 |
| D2.3 洞察深度 | 提炼本质规律，而非情绪噪音或正确废话 | 1-3 只有摘要/口号；4-6 有 takeaway 但浅；7-8 能形成判断框架；9-10 能产出可复用产品/面试框架 | PM takeaway、方法论、反例 |
| D2.4 AI PM 目标对齐 | 服务面试、作品集、公司研究、产品判断 | 1-3 与目标脱节；4-6 泛 AI 有用但不够岗位化；7-8 明确服务 AI PM；9-10 可直接支撑目标公司面试/项目 | 目标场景字段或显式说明 |
| D2.5 知识分层分类型 | 学术论文/官方文档/经验认知/数据榜单/新闻有定制选择标准 | 1-3 所有知识同质处理；4-6 有粗分；7-8 每类有准入标准；9-10 每类有准入、保鲜、呈现和拒绝规则 | frontmatter、SOP、目录结构 |

10 分锚点：行业专家盲评认为选材服务目标明确，且每类知识的收录/拒绝标准可复用、可审计。

### D3 · 结构与组织

| 子指标 | 定义 | 计分规则 | 证据要求 |
|---|---|---|---|
| D3.1 MECE 与层级 | 主题边界清楚，目录能覆盖目标范围且不严重重叠 | 1-3 无清晰目录；4-6 有目录但交叉/空白明显；7-8 层级稳定；9-10 支持新增内容自动路由 | 目录、文件树、被错分/漏分样例 |
| D3.2 双链/反链/标签健康 | 文档之间可点击互链，标签受控，反链能支持维护 | 1-3 无链接；4-6 有零散链接；7-8 双链与标签可用；9-10 有反链统计与同义标签治理 | Obsidian graph、frontmatter、标签清单、死链报告 |
| D3.3 演化链与时间线 | 理论/产品/公司变化能看到前后关系 | 1-3 静态堆叠；4-6 有日期但无对照；7-8 有新旧对照；9-10 可按主题生成演化时间线 | timeline、changelog、历史段落、迁移说明 |
| D3.4 多入口导航 | 支持按学习路径、面试场景、目标公司、知识类型进入 | 1-3 只有文件列表；4-6 有总目录；7-8 多入口可用；9-10 入口能反映用户真实任务 | 总目录、场景页、公司页、学习路径 |
| D3.5 人物/公司/概念索引 | 关键人物、公司、产品、概念是一等索引对象 | 1-3 无索引；4-6 零散出现；7-8 有索引页；9-10 能聚合同一对象的观点/事件/链接 | 人物页、公司档案、概念页、索引链接 |

10 分锚点：任一核心概念可从索引、标签、反链或人物/公司入口在 ≤2 跳内定位；历史演化与当前主路径分离。

### D4 · 呈现与体验

| 子指标 | 定义 | 计分规则 | 证据要求 |
|---|---|---|---|
| D4.1 纪要质量 | 摘要能提炼结论、证据、适用边界和行动意义 | 1-3 复制粘贴；4-6 有摘要但重点散；7-8 重点清楚；9-10 可直接用于复习/面试 | 摘要样本、PM takeaway、删减前后对比 |
| D4.2 形式适配 | 按内容选择文本、表格、图、链接短评、引文、资源索引 | 1-3 全是文本墙；4-6 偶有表格；7-8 形式基本匹配；9-10 三粒度策略稳定执行 | 条目类型、表格/图表、链接短评样例 |
| D4.3 数据图表口径 | 图表与数字有来源、日期、样本和不可比说明 | 1-3 无图表或无源数字；4-6 有图表但口径弱；7-8 口径完整；9-10 图表可复用到面试材料 | 图表、source_url、口径说明、accessed_at |
| D4.4 阅读深度分层 | 同一主题有速查、正常阅读、深读路径 | 1-3 只有长文；4-6 有目录但无速查；7-8 有摘要/正文/深读链接；9-10 可按时间预算阅读 | TLDR、正文、references、reading path |
| D4.5 工具特性利用 | Markdown/Obsidian 特性用于导航、callout、标签、任务、引用 | 1-3 基本文本；4-6 少量 callout；7-8 特性稳定使用；9-10 工具特性服务自动化体检 | frontmatter、callout、tasks、tags、graph |

10 分锚点：速查、正常阅读、深读三种模式都舒适；关键对比优先图表化且口径完整。

### D5 · 可维护性与流程

| 子指标 | 定义 | 计分规则 | 证据要求 |
|---|---|---|---|
| D5.1 git/版本/回滚 | 所有重要变更可追溯、可 diff、可回滚 | 1-3 无版本；4-6 有 git 但提交粗；7-8 原子提交；9-10 每轮有 tag/rollback 指令 | git log、tag、commit diff、rollback 文档 |
| D5.2 changelog/diff 可观测 | 人能看懂每次改了什么、为什么改 | 1-3 无记录；4-6 人工流水账；7-8 changelog 与 diff 对齐；9-10 dashboard 可浏览变更 | CHANGELOG、run report、dashboard |
| D5.3 SOP 遵守 | 核心流程成文且实际被运行引用 | 1-3 无 SOP；4-6 有 SOP 但不执行；7-8 运行报告引用 SOP；9-10 事故自动反哺 SOP | SOP 文件、运行记录、事故复盘 |
| D5.4 自动化管线与 HITL 边界 | 自动收集/诊断/更新，同时明确人工把关点 | 1-3 全手工；4-6 自动化零散；7-8 有稳定管线；9-10 source funnel 与人审策略可观测 | 脚本、调度、source funnel、确认记录 |
| D5.5 飞轮数据与坏 case 回归 | 人类覆写、失败样本、平台反馈能变成下一轮数据 | 1-3 无飞轮；4-6 有记录但不进入 eval；7-8 坏 case 入 regression；9-10 覆写驱动 prompt/SOP/rubric 更新 | flywheel 数据、regression 集、覆写记录 |
| D5.6 运行报告完整性 | 每轮报告输入、输出、评分、归因、成本、下一步 | 1-3 只给结果；4-6 有摘要；7-8 报告字段完整；9-10 可直接复现实验 | manifest、results.jsonl、成本字段、归因 |

10 分锚点：每轮运行有输入、输出、评分、归因、成本、人工覆写和回滚点；历史失败自动进入回归集。

### D6 · 标杆实践采纳率

使用 `research/golden-kb-study/12-D6对标checklist.md`。D6 分数 = `1 + 9 × (达成得分 / 适用项总数)`。

## 4. Behavior Eval

| 维度           |  权重 | 定义                               |
| ------------ | --: | -------------------------------- |
| B1 路由与判断正确性  | 35% | 收/拒/候选/核实/归档/改写/拆分决策             |
| B2 处理质量      | 25% | 纪要、结构、信息形态、目标对齐                  |
| B3 溯源与风险纪律   | 20% | 一手源、日期、事实/观点区分、版权、隐私、平台风险        |
| B4 边界与升级意识   | 10% | 何时问用户、何时 HITL、何时承认不确定            |
| B5 过程记录与可复用性 | 10% | 是否留下 changelog、SOP 更新、坏 case、回归项 |

### B1 可辩护性通道

参考行为不是唯一答案。若 actor 与 gold 不一致，Judge 不得直接判错，而要评估理由质量：

- 证据是否充分
- 是否引用 SOP/SL 判据
- 是否识别风险
- 是否说明替代路线
- 是否可回滚

理由充分且风险可控，可得 7+；踩 auto-fail 仍不得超过 3。

## 5. Split 与防过拟合纪律

- smoke 15：每轮必跑。
- regression 15：每轮必跑，含历史失败。
- holdout 10：每 3 个 agent prompt 版本跑一次，结果不得用于直接调 prompt。
- full-only 60：阶段里程碑跑。
- 每轮追加 3-5 条当周早报新鲜条目作 live set，单列得分。
- 训练侧（smoke/regression）与 holdout 分差 > 1.0，dashboard 触发过拟合警报。

## 6. Judge 校准

每轮 Judge 正式评分前，先跑 `evals/judge-calibration-set.md`。用户确认的标准分作为 gold calibration。

- 平均偏差 <= 0.5：正常。
- 0.5 < 平均偏差 <= 1.0：低风险，报告中标注。
- 平均偏差 > 1.0：本轮全部评分标记低置信，需复评或调整 Judge prompt。

校准集由用户最终确认。未确认前，只能作为候选校准集。

## 7. 效率门槛（不入权重，一票项）

每轮必须报告：tokens、运行时长、工具调用次数、用户确认请求数、用户把关耗时。

默认预算：

| 运行类型 | 预算 |
|---|---|
| smoke 15 | <= 45 分钟；用户确认 <= 3 次 |
| smoke+regression 30 | <= 90 分钟；用户确认 <= 6 次 |
| state Doctor | <= 2.5 小时；用户 review <= 45 分钟 |
| full milestone | 需提前声明预算并经用户确认 |

超预算且未提前说明 = 该轮 cost-fail，不得以高分通过。

## 8. 状态↔行为联动

连续两轮行为分上升但状态分不动，标记为“无效运行”：agent 会处理单条 case，但没有让知识库变好。必须归因到 prompt、SOP、写回机制或状态 eval 缺口。
