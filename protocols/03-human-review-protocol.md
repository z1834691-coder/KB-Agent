# 人把关协议 v0.1

## 目标

让 KB-Agent 自动化尽可能多的重复劳动，同时在低置信、高风险、价值判断、不可逆操作上及时停下来问用户。

## 用户已确认参与的环节

用户已经明确会参与：

1. 评分标准设计与核查。
2. baseline 判断。
3. 数据集建设与选择。
4. 第一轮 Judge 校准集打分。
5. 每轮 eval 后抽查高分 case 和低分 case，校准 Judge 是否乱打。
6. Agent 推出并正常运行后，对低置信度、信息冲突、版权敏感、删除内容等动作把关。
7. 第一轮针对 baseline 的大改是否采纳，由用户确认。

## 人审等级

| 等级 | 名称 | 是否阻塞 | 示例 |
|---|---|---|---|
| H0 | 无需人审 | 不阻塞 | 只读检查、dashboard build、schema 校验 |
| H1 | 事后汇报 | 不阻塞 | 小修 typo、补 source metadata、生成报告 |
| H2 | 批准后执行 | 阻塞 | 大改结构、移动文件、改 rubric、导入大量新内容 |
| H3 | 用户亲自裁判 | 阻塞 | Judge 校准、baseline 大修采纳、争议事实裁决 |
| H4 | 禁止自动化 | 阻塞且默认拒绝 | 发布到公网、删除大量内容、处理密钥、公开敏感个人信息 |

## 必须问用户的触发条件

### 内容风险

- 低置信度但影响主知识库结论。
- 信息源冲突，且无法通过一手源解决。
- 涉及目标公司负面评价或求职风险。
- 涉及版权敏感内容：公众号全文、付费内容、截图、长引文。
- 涉及隐私：简历、面试、私信、个人经历、API key。

### 结构风险

- 删除内容。
- 批量重命名、移动、拆分、合并文件。
- 新增/删除一级目录或核心文档。
- 改变知识库主路径或总览结构。

### Harness 风险

- 修改 rubric 权重。
- 修改 challenge dataset gold。
- 修改 holdout 或泄露 gold 给 actor。
- 校准集标准分未确认却用于正式评分。
- eval 高分但证据不足。

### 运行风险

- 工具需要网络或权限升级。
- 预计超出效率预算。
- 需要公开 GitHub 或发布外部内容。

## 人审输出格式

当需要用户判断时，agent 必须给出：

```markdown
## 需要你确认

动作：
风险：
证据：
推荐选择：
备选选择：
不做的后果：
可回滚方式：
```

避免问“你怎么看”。问题必须具体、可决策。

## Eval 后的人审流程

每轮 eval 后：

1. agent 自动列出最高分 3 个 case、最低分 3 个 case、分歧最大 case。
2. 用户抽查这些 case 的评分是否符合直觉。
3. 若用户覆写分数，记录为 calibration/flywheel 数据。
4. 如果 Judge 与用户平均偏差 > 1.0，下一轮评分标记低置信。

## 第一轮 baseline 大改协议

针对 baseline-v0 的第一次大修必须先给用户：

- 大修范围
- 预计影响文件
- 每项改动的目标维度
- 是否可回滚
- 预期提分

用户确认后才执行。执行时每个逻辑改动独立 commit。

## 正常运行后的自动化边界

可以自动做：

- 小范围 source metadata 补全。
- 低风险链接修复。
- 候选池整理。
- 生成更新报告和 dashboard。
- 把坏 case 写入 regression 候选。

必须问用户：

- 低置信事实写入主文档。
- 删除或大规模移动内容。
- 版权敏感摘要/转载处理。
- 与目标公司相关的主观判断。
- 公开发布或开源前最终确认。

## 人审数据也要进入飞轮

每次人审都记录：

```yaml
review_id: ""
trigger: ""
agent_recommendation: ""
user_decision: ""
reason: ""
should_update_prompt: true | false
should_update_sop: true | false
should_update_rubric: true | false
```

用户的覆写是最高价值数据，优先用于修 prompt、SOP 和 Judge 校准。
